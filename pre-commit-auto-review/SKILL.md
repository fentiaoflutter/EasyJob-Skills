# 提交前自动审查

> 一句话目标：在 `git commit` 之前先跑一轮结构化审查，减少"能编译但不可交付"的提交。

## 触发场景

用户这样说时调用：

- 「帮我提交」「准备 commit」「提交前看一下」「review 一下这些改动」「提交前检查」
- 项目里如果配了拦截 hook：用户直接 `git commit` 被拦下来需要先走审查。

## 执行流程

### 1. 读取变更全貌

| 命令 | 用途 |
|------|------|
| `git status` | 改动范围 |
| `git diff` | 未暂存改动 |
| `git diff --cached` | 已暂存改动 |
| `git log -5 --oneline` | 沿用仓库 commit 风格 |

### 2. 风险审查（先高后低）

| 维度 | 关注点 |
|------|--------|
| 正确性 | 逻辑错误、空指针 / 边界、状态不同步、异步时序 |
| 回归风险 | 是否误改无关代码、是否改变旧行为、兼容性影响 |
| 安全与隐私 | 敏感日志、密钥、用户信息泄露 |
| 可维护性 | 命名、重复逻辑、魔法值、注释 ↔ 意图是否匹配 |
| 测试与验证 | 是否缺最小验证步骤；必要时建议补测试点 |

### 3. 结论输出

- 有阻断项 → 先列阻断项，**禁止直接提交**。
- 无阻断项 → 给"可提交"结论 + 最小验证清单 + commit message 草案（强调 why）。

### 4. 用户确认后再提交

- 用户没明确说"提交"时，只做审查，**不自动 commit**。
- 明确要求提交时，再执行 add / commit 并回报结果。

## 输出模板

```markdown
提交前审查结果：

- 结论：可提交 / 暂不建议提交
- 阻断问题：
  - [若无则写"无"]
- 主要风险提醒：
  - ...
- 最小验证清单：
  - [ ] 场景 A
  - [ ] 场景 B
- 建议 commit message：
  - type(scope): ...
    ...
```

## commit message 规则

- 类型优先用：`feat` / `fix` / `refactor` / `docs` / `test` / `chore`。
- 首行简洁，聚焦变更**意图与收益**。
- 正文 1-2 句说明背景与影响范围。

## 禁止事项

- 未经用户明确允许，不直接 `git commit` / `git push`。
- 不把 `.env` / 密钥 / 凭证文件加入提交。
- 不为"凑提交"创建空提交。
- 跨多个模块的改动，审查时关注跨模块一致性，不放行只改一半的状态。

## 与 Hook 联动（可选）

如果你在编辑器（如 Cursor）里配了 `git commit` 拦截 hook，可在 hook 脚本中要求先调用本 skill 才能放行；放行方式按 hook 脚本约定（如设置 `PRECOMMIT_REVIEW_OK=1`）。

## 示例

> **用户**：帮我提交一下，commit message 你来写。
> **Skill**：跑 status / diff / log → 跨 5 维度审查 → 输出"可提交 + 最小验证清单 2 项 + `fix(scope): ...` 草案" → 等用户确认后再执行 `git add` / `git commit`。
