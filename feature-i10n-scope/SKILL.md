# 需求范围文案（I10n）技能

> 一句话目标：在**用户明确给出的需求范围内**完成文案的补充、整理、一致性检查，让代码 ↔ 主翻译文件 ↔ 其它语种占位三者对齐。

## 触发场景

用户这样说时调用：

- 「帮我检查 PR-XXXXX 的文案 / 国际化」「补一下这个目录下的翻译 key」「这次需求文案有没有漏 zh_CN」
- 用户提供 PR 编号 / 分支名 / 路径前缀 / 文件列表，要求在该范围内查 i10n。

## 范围输入（必须先定界）

执行前向用户确认或从上下文读取**至少一项**，未指定先提问再动手：

| 输入类型 | 示例 | 定界方式 |
|----------|------|----------|
| PR / 需求编号 | `PR-XXXXX`、`XXXXX` | `rg -i 'pr[_-]?0*XXXXX\|XXXXX'` 于仓库；再读命中文件 |
| 分支名 | `dev/feature/pr_XXXXX` | `git diff origin/dev_v*...HEAD --name-only` |
| 路径前缀 | 例 `<main-module>/lib/page/foo/bar/` | 仅检索该目录及子目录 |
| 文件列表 | 用户粘贴 paths | 仅处理所列文件 |

**禁止**：在未定界时对全仓库做文案扫描（除非用户明确要求全量）。

## 项目约定（按你的栈填写）

不同栈的翻译机制对应关系：

| 概念 | Flutter / GetX | React (react-intl / i18next) | Android 原生 | iOS 原生 |
|------|----------------|------------------------------|--------------|----------|
| 翻译 key 调用 | `'key'.tr` / `'%s'.trArgs([...])` | `t('key')` / `<FormattedMessage id="key" />` | `getString(R.string.key)` | `NSLocalizedString("key", ...)` |
| 主中文文件 | `<main-module>/assets/I10n/zh_CN.json` | `locales/zh-CN.json` | `res/values-zh/strings.xml` | `zh-Hans.lproj/Localizable.strings` |
| 其它语种 | `assets/I10n/generated/<locale>.json` | `locales/<locale>.json` | `res/values-<locale>/strings.xml` | `<locale>.lproj/Localizable.strings` |
| 加载入口 | `TranslationService` | `i18n.init({...})` / `I18nextProvider` | Android 资源系统 | iOS 资源系统 |

通用约定：

- **优先用简体中文 key**（与主翻译文件中条目一致）
- 触及历史模块的前缀 key（如 `xx_` 系列）时保持原风格，不强行改

子包若独立 i10n，路径改为对应子包的翻译目录，步骤相同。

## 豁免清单（不当硬编码处理）

下列情形视为正常实现，**报告里仅在末尾用一行「豁免（正常）」归类**，不要逼用户改成可翻译 key：

| 类别 | 说明 | 示例 |
|------|------|------|
| 空值 / 占位符 | 列表或详情无数据时占位 | `'--'`、`'—'` |
| 接口 / 后端直出 | 多语言由后端约定；待接口确认枚举的也不强推 | 字段直接展示 |
| 国际通用符号 | 稳定计价币种 / ticker 字面量 | 货币符号、单位代码 |
| 纯数据拼接 | 数字、百分比、精度格式化 | `0.02%`、`1.5 <unit>` |
| 调试 / 日志 | 开发期日志文案 | `debugPrint` / `console.log` / `Log.d` |

> **与"硬编码问题"的界限**：用户可见的**完整中文句子**（标题、说明、按钮、Toast）必须走翻译机制；豁免清单只覆盖上表及**同类**情形。

## 执行流程

### 1. 收集需求相关文件集合

用 PR 注释 / 分支 diff / 用户给定路径得到**改动文件列表**（按栈对应：`.dart` / `.ts(x)` / `.kt` / `.swift` / ...）。

### 2. 从代码抽"用户可见文案"

在已定界文件内检索：

- **已国际化**：对应栈的翻译调用（`.tr` / `t(...)` / `getString(...)` / `NSLocalizedString(...)`）
- **疑似硬编码**：UI 参数中是**自然语言**且**未走翻译机制**的字面量（典型位置：`Text(` / `Toast` / `SnackBar` / `Semantics` / `title:` / `hint:` / `label:` / JSX 子节点字符串 / `TextView.setText(...)` / `UILabel.text = ...`）
- **占位 / 数据**：命中豁免清单的项**不**计入"待修复"

### 3. 与主翻译文件对照

对每个翻译 key：

- **必须在**主翻译文件中存在；缺失则补一条（value 与产品 / 设计稿一致）
- **带参数占位**：确认中文 value 中含 `%s` / `@param` / `{name}` 等与所用框架约定一致的占位
- **重复 key**：同语义不要拆多个近义 key；已有同义 key 优先复用（仅在本次需求范围内改）

### 4. 整理（可选但建议）

- 新增条目集中在主翻译文件尾部，或按模块注释分组（跟随项目既有习惯）
- 同需求新增多条 key，附**清单表**：key | 中文 value | 出现文件

### 5. 其它语种

- 要求"一并更新外语"：在对应其它语种文件为每个新 key 补英文（或占位），说明"正式翻译走你项目的翻译平台"
- 仅要求中文交付：明确列出新增 key 列表供后续导入翻译平台

## 输出报告（必须）

```markdown
- 范围：PR / 路径 / 文件数
- 新增 / 修改的 JSON 条目：
  | key | 中文 value | 出现文件 |
  ...
- 硬编码问题（不在豁免清单内）：
  - 文件:行 → 建议改为 key 名 + 翻译调用
- 豁免（正常）：含 ...（对应豁免表类别）
- 验证方式：切语言跑一遍相关页面；或运行该栈的静态分析（如 `flutter analyze` / `tsc --noEmit` / `lint`）
```

## 禁止事项

- 不要在未定界时全仓库扫描 / 重排主翻译文件。
- 不要把长说明文拆成大量近义短 key 且无复用。
- 不要只改代码翻译调用而不同步主翻译文件（反之亦然）。
- 日志 / Toast 文案**不要**写 token、完整手机号等敏感信息。

## 示例

> **用户**：检查一下 PR-12345 的文案，有没有漏掉的 i10n。
> **Skill**：`git diff` 拿到改动文件 → 抽翻译调用与硬编码 → 对照主翻译文件补 3 条新 key → 报告里列「新增 3 条 + 硬编码 1 处 + 豁免（占位符 / 后端字段）」+ 验证步骤。
