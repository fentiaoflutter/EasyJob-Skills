# 需求范围文案（I10n）技能

> 一句话目标：在**用户明确给出的需求范围内**完成文案的补充、整理、一致性检查，让代码 ↔ `zh_CN.json` ↔ 其它语种占位三者对齐。

## 触发场景

用户这样说时调用：

- 「帮我检查 PR-XXXXX 的文案 / 国际化」「补一下这个目录下的 `.tr`」「这次需求文案有没有漏 zh_CN」
- 用户提供 PR 编号 / 分支名 / 路径前缀 / 文件列表，要求在该范围内查 i10n。

## 范围输入（必须先定界）

执行前向用户确认或从上下文读取**至少一项**，未指定先提问再动手：

| 输入类型 | 示例 | 定界方式 |
|----------|------|----------|
| PR / 需求编号 | `PR-XXXXX`、`XXXXX` | `rg -i 'pr[_-]?0*XXXXX\|XXXXX'` 于仓库；再读命中文件 |
| 分支名 | `dev/feature/pr_XXXXX` | `git diff origin/dev_v*...HEAD --name-only` |
| 路径前缀 | 例 `<main-module>/lib/page/foo/bar/` | 仅检索该目录及子目录 |
| 文件列表 | 用户粘贴 paths | 仅处理所列文件 |

**禁止**：在未定界时对全仓库做文案扫描（除非用户明确要求全量）。

## 项目约定（按你的项目实际填写）

> 下面以「主模块 `<main-module>` + 若干子包」的常见结构为例，迁移时按真实目录替换。

| 项 | 值 |
|----|-----|
| 主中文包 | `<main-module>/assets/I10n/zh_CN.json` |
| 其它语种 | `<main-module>/assets/I10n/generated/<locale>.json`（翻译平台 / 脚本同步） |
| 加载入口 | `<main-module>/lib/I10n/translation_service.dart` |
| 代码用法 | GetX `'key'.tr` / `'带%s'.trArgs([...])` |
| Key 规范 | **优先用简体中文做 key**；触及历史模块的 `xx_` 前缀 key 时保持原风格，不强行改 |

子包若独立 I10n（如 `<sub-module-a>` / `<sub-module-b>` / `<sub-module-c>` / `<components-pkg>`），路径改为对应包下的 `assets/I10n/`，步骤相同。

## 豁免清单（不当硬编码处理）

下列情形视为正常实现，**报告里仅在末尾用一行「豁免（正常）」归类**，不要逼用户改成 `.tr`：

| 类别 | 说明 | 示例 |
|------|------|------|
| 空值 / 占位符 | 列表或详情无数据时占位 | `'--'`、`'—'` |
| 接口 / 后端直出 | 多语言由后端约定；待接口确认枚举的也不强推 | 字段直接展示 |
| 国际通用符号 | 稳定计价币种 / ticker 字面量 | 货币符号、单位代码 |
| 纯数据拼接 | 数字、百分比、精度格式化 | `0.02%`、`1.5 <unit>` |
| 调试 / 日志 | `debugPrint` 等开发期文案 | — |

> **与"硬编码问题"的界限**：用户可见的**完整中文句子**（标题、说明、按钮、Toast）必须走 `.tr` + JSON；豁免清单只覆盖上表及**同类**情形。

## 执行流程

### 1. 收集需求相关文件集合

用 PR 注释 / 分支 diff / 用户给定路径得到 **Dart 文件列表**。

### 2. 从代码抽"用户可见文案"

在已定界文件内检索：

- **已国际化**：`'...'.tr`、`"...".tr`、`trArgs`、`trPlural`
- **疑似硬编码**：`Text(` / `AppToast.show(` / `SnackBar` / `Semantics(label:)` / `title:` / `hint:` / `label:` 等参数中是**自然语言**且**无 `.tr`** 的字面量（排除豁免清单）
- **占位 / 数据**：命中豁免清单的项**不**计入"待修复"

### 3. 与 `zh_CN.json` 对照

对每个 `.tr` / `.trArgs` 的 key：

- **必须在** `zh_CN.json` 中存在；缺失则补一条（value 与产品 / 设计稿一致）
- **trArgs 占位**：确认中文 value 中含 `%s` / `@param` 等 GetX 约定占位
- **重复 key**：同语义不要拆多个近义 key；已有同义 key 优先复用（仅在本次需求范围内改）

### 4. 整理（可选但建议）

- 新增条目集中在 JSON 文件尾部，或按模块注释分组（跟随项目既有习惯）
- 同需求新增多条 key，附**清单表**：key | 中文 value | 出现文件

### 5. 其它语种

- 要求"一并更新外语"：在对应 `generated/*.json` 为每个新 key 补英文（或占位），说明"正式翻译走你项目的翻译平台"
- 仅要求中文交付：明确列出新增 key 列表供后续导入翻译平台

## 输出报告（必须）

```markdown
- 范围：PR / 路径 / 文件数
- 新增 / 修改的 JSON 条目：
  | key | 中文 value | 出现文件 |
  ...
- 硬编码问题（不在豁免清单内）：
  - 文件:行 → 建议改为 key 名 + `.tr`
- 豁免（正常）：含 ...（对应豁免表类别）
- 验证方式：切语言跑一遍相关页面；或 `flutter analyze` 相关包
```

## 禁止事项

- 不要在未定界时全仓库扫描 / 重排 `zh_CN.json`。
- 不要把长说明文拆成大量近义短 key 且无复用。
- 不要只改代码 `.tr` 而不同步 `zh_CN.json`（反之亦然）。
- 日志 / Toast 文案**不要**写 token、完整手机号等敏感信息。

## 示例

> **用户**：检查一下 PR-12345 的文案，有没有漏掉的 i10n。
> **Skill**：`git diff` 拿到改动 Dart 文件 → 抽 `.tr` 与硬编码 → 对照 `zh_CN.json` 补 3 条新 key → 报告里列「新增 3 条 + 硬编码 1 处 + 豁免（占位符 / 后端字段）」+ 验证步骤。
