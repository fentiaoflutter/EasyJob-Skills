# 接口接入工作流（Flutter / GetX）

> 一句话目标：把"用户给的接口文档"按 API → Model → Controller → Page 顺序稳定接入，减少字段类型错误、解析不兼容、筛选交互不一致和联调反复。

## 触发场景

用户这样说时调用：

- 「我发接口文档给你，帮我接入」「继续接入这个接口」「按文档改字段」「联调修复」
- 用户贴出 request / response 样例日志，要求核对字段与展示。

## 输入约定

- 必备：接口文档（路径、方法、参数、响应结构、枚举）。
- 推荐：一份真实请求 / 响应样例日志（用于校对）。
- 缺失文档时：先索要再动手，不要从代码反推规格。

## 执行流程

### 1. 文档抽取与确认

- 提取：路径、方法、请求参数（必填 / 选填 / 类型）、响应结构、枚举与特殊规则。
- 若文档与历史实现冲突，按 **「文档 + 实际返回」** 收口。

### 2. 最小改动接入（四层）

| 层 | 文件示例 | 关注点 |
|----|----------|--------|
| API | `<biz>_api.dart`（必要时含 `<biz>_api.g.dart`） | 路径、方法、参数命名严格对齐文档 |
| Model | 实体类 | 字段名 / 类型与文档一致；必要时做兼容解析（如 `list` vs `<resource>List`） |
| Controller | 业务 controller | 请求参数、筛选触发刷新、分页字段 |
| Page | 页面 / Widget | 只改本需求需要的展示与交互 |

### 3. 兼容与风险控制

- 数字 / 字符串混用字段做安全转换（`_toInt` / `_toString` 思路）。
- 对偶发空值、`null`、字段别名做最小兼容。
- **禁止顺手改无关模块**；发现无关问题先告知用户再处理。

### 4. 联调校验（基于日志）

- 用 [`clean-log`](../clean-log) 把日志清成"干净 JSON（含 URL）"。
- 核对清单：
  - [ ] 请求参数名完全一致（大小写严格，例 `pageSize` vs `pagesize`）
  - [ ] 字段类型与文档一致（尤其 `String` / `int`）
  - [ ] 响应字段被正确解析并落到 UI
  - [ ] 分页不重复、筛选变更触发刷新
  - [ ] 特殊冲突规则已实现（如"状态筛选优先于隐藏开关"）
  - [ ] UI 文案与设计一致（前缀、换行、隐藏规则）

### 5. 提交策略

- 按需求粒度提交，**不带无关文件**。
- 误改无关文件先恢复再提交。
- 提交信息聚焦 why + scope，如 `fix:` / `feat:`。

## 输出模板

```markdown
接口接入完成：

- 接口：`METHOD /path`
- 主要改动：
  - API: ...
  - Model: ...
  - Controller: ...
  - Page: ...
- 文档对齐点：
  - 参数: ...
  - 字段类型: ...
  - 特殊规则: ...
- 联调结果：
  - 请求/响应核对：通过 / 问题点 ...
  - 剩余风险：...
```

## 禁止事项

- 不从代码反推接口规格（必须以文档为准）。
- 不顺手改无关模块。
- 不把无关文件带进 commit。

## 示例

> **用户**：这是接口文档，帮我接入到优惠券记录页。
> **Skill**：抽文档参数 / 字段 → API/Model/Controller/Page 四层最小改 → 用真实日志核对 6 项 checklist → 按需求范围提交，回报 fix scope 与剩余风险。
